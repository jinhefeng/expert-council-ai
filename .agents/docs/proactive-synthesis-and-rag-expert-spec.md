# 主动流式主持人与 RAG 知识外挂及信息索取环《功能与交互说明书》

本文档基于“人类主导、AI 深度参与引导评审”的设计哲学，定义了 `Design Council AI` 系统中外部 RAG 数据库对接、流式主动引导主持人、以及人机信息索取环的业务交互与技术规范。

---

## 一、 功能 A：RAG 专家私有知识库外挂 (External RAG Database Integration)

### 1.1 业务背景
若直接将长篇参考规范（如几百KB的安全准则）全文注入专家的系统提示词，会导致大模型上下文（Context Window）爆炸，不仅带来高昂的 Token 费用，还会引发“Lost in the Middle”的注意力分心效应。因此系统不采用本地文本暴力注入，而应当直接**对接外部已有的 RAG 知识库数据库**。

### 1.2 检索架构与数据模型
系统支持将专家定义与外部向量知识库 API 连接：
1. **数据模型扩展**：
   ```typescript
   export type Expert = TenantScoped & {
     // ... 原有字段
     ragEnabled?: boolean;     // 是否启用外部 RAG 知识检索
     ragEndpoint?: string;     // 外部 RAG 接口地址 (例如 Dify/FastGPT/自研向量服务 API)
     ragToken?: string;        // 外部 RAG API 鉴权 Token
     ragDatasetId?: string;    // 目标知识库/数据集 ID (Collection ID)
   }
   ```
2. **检索流程（API 对接）**：
   - 当触发该专家发言时，后端 API 检测到 `ragEnabled === true` 且配置了 `ragEndpoint`。
   - 向 `ragEndpoint` 发送 HTTP POST 请求，载荷中包含当前会议议题 `{ query: question, datasetId: ragDatasetId }` 并带上 `ragToken` 鉴权首部。
   - 外部 RAG 数据库进行高性能向量计算并返回 Top-K 相关的**文本语义切片（Chunks）**。
   - 后端拼装：将检索到的最相关内容作为客观事实背景，拼接进大模型输入：
     ```text
     【从外部RAG知识库检索到的相关参考规范】：
     {retrievedChunks}
     ```
3. **未配置处理**：若用户未指定外部 API，或外部 API 连接超时，系统不执行任何本地文件降级，直接返回空字符串，专家仅根据原生的 System Prompt 正常发言。

## 二、 功能 B：基于“反馈环状态机”的主动引导与自主决策 (Feedback Loop State Machine)

### 2.1 第一性原理与统一模型定义
从会议评审的本质来看，无论哪种讨论模式，其底层都遵循同一个**“发表 ➔ 冲突 ➔ 局部决议 ➔ 反馈论证 ➔ 一致收敛”**的反馈环路（Feedback Loop）。

系统在全站各处统一以下三态名词：
- **被动传统 (`passive`)**：传统被动总结。
- **协调引导 (`facilitative`)**：人类决策主导，AI 辅助推动（**创建会议时的默认模式**）。
- **自主决策 (`autonomous`)**：AI 自主决策仲裁。

主持模式与信息追问之间存在**联动锁定关系**：
- **连锁限制**：当主持模式设定为 **自主决策 (`autonomous`)** 时，信息索取追问环将被**强制关闭且禁用**，且大循环中强制置为 `false`，确保在自主模式下绝不会发出澄清索取。

---

### 2.2 传统被动模式 (Passive Traditional Mode)
- **在统一状态机中的流控表现**：
  1. 系统不更新 `currentDecisions`。
  2. 当一个专家发言完毕后，流控抉择中枢检测剩余候选专家列表 `remainCandidates`：
     - **若非空**：指派下一个专家发言，继续 while 循环。
     - **若为空**：触发主持人（AI）流式进行一次性终极总结（Synthesis）。
  3. 主持人流式总结结束后，直接跳出 while 循环，自动调用最终结论提炼并归档。

---

### 2.3 协调引导模式 (Human-Steered Facilitative Mode)
- **在统一状态机中的流控表现**：
  1. 每轮讨论（例如 2 位专家发言完毕）后，大模型输出本轮共识总结。
  2. 弹出**“决策协调看板”**。人类可手动选择选项或填写补充意见。
  3. 人类点击【🚀 提交并进入下一轮讨论】将作为新输入，返回最前置的澄清判定（Step 1），然后继续开启下一轮。
  4. 人类点击【🏁 达成共识，结案陈词】将直接退出循环并自动触发提炼结论。

## 三、 功能 C：主持人与决议卡片数据独立隔离 (Moderator Data Separation)

为了实现系统数据模型的优雅高内聚，主持人的综合决议数据将不再塞入专家专属的 `expertStance` 属性中，而是统一存盘在专用的 `moderatorSummary` 属性下：

```typescript
export type ChatMessage = TenantScoped & {
  // ... 原有字段
  role: "user" | "expert" | "moderator";
  moderatorSummary?: {
    consensus: string[];       // 总结共识
    disagreements: string[];   // 主要分歧
    decisions: string[];       // 最终决策
    nextActions: string[];     // 下一步行动
  };
};
```

流式主持人总结在完成 SSE 数据流拼接后，将利用 `cleanAndParseJson` 提取尾部的 JSON 并落盘至 `moderatorSummary`，同时正文文本中将抹除 JSON 残留，确保卡片与气泡渲染整洁美观。

## 四、 功能 D：前置信息澄清与追问环 (Clarification & Inquiry Loop)

### 4.1 会议控制开关与前置流程
- **开关设计**：在会议设置中，新增 `enableInquiryLoop?: boolean` 开关。
- **澄清校准逻辑**：
  1. **开关启用 (`enableInquiryLoop === true`)**：
     * **Step 1**：当人类发送新的消息（包括驳回整改等）或在决策面板上选择了下一步意见后，立即进入本流程。本流程出现在人类增加信息之后，用于对输入信息进行澄清与校准。
     * **Step 2**：主持人进入追问判定模式（调用后端接口），根据后台可配置的 `inquiryJudgmentPrompt` 检查当前信息是否足够。
     * **Step 3**：若判断需要，输出包含 `<inquiry>问题内容</inquiry>` 标签。系统挂起事件循环，并在当前消息流下方展示**“信息补充面板”**。
     * **Step 4**：人类可以选择「跳过」或「补充信息并提交」。
     * **Step 5**：若用户选择「跳过」，则本澄清流程结束，正常推进专家论证。若用户补充了信息并提交，信息将被拼入上下文背景，系统自动**返回 Step 1** 重新进行信息澄清判定，以此往复。
  2. **开关关闭 (`enableInquiryLoop === false`)**：直接跳过澄清追问阶段，正常推进专家论证。

## 五、 主持人决策可选项与决策面板 (Moderator Decision Options)

### 5.1 决策面板与主持模式流转
- **流程规范**：
  1. **选项生成**：专家发言完毕，主持人流式总结完成后，系统根据后台可配置的 `decisionOptionsPrompt` 自动请求大模型，生成 2-4 个关于下一步论证的方向性意见备选项，并存入状态。
  2. **面板展示**：决策面板在**主持人总结完成之后**才会出现。
  3. **人类与 AI 协作流控**：
     - **被动传统 (`passive`)**：不进入决策流程。主持人总结完，直接跳出讨论循环，并**自动触发提炼结论动作**，完成结案归档。
     - **协调引导 (`facilitative`)**：展现包含这些备选项的卡片列表，并提供一个“手写补充额外意见”输入框。
       - 用户点击【🚀 提交并进入下一轮讨论】将作为新输入重新进入讨论循环。
       - 用户点击【🏁 达成共识，结案陈词】将跳出讨论循环，并**自动触发提炼结论动作**，完成结案归档。
     - **自主决策 (`autonomous`)**：由 AI 主持人代替人类自动从备选项中挑出一个，作为新输入。系统自动返回信息追问流程的 Step 1。最多运行 3 轮强制收敛，收敛后自动跳出讨论循环，并**自动触发提炼结论动作**，完成结案归档。

### 5.2 流程规范与限制优化
1. **前置轮次研判**：专家发言完毕、主持人总结完成后，系统应**立即检验当前已进行的轮次 `currentRound`**。如果已达到最大自主轮次上限（`currentRound >= maxAutonomousRounds`，默认 3 轮），系统应**直接抛出收敛提示文案并结束讨论**，自动触发提炼结论。严禁先展现决策面板或卡片给用户进行选择，避免用户体验受挫。
2. **精确干预文案指示**：在决策面板流转返回时，传回的 action 对象须明确携带 `isAuto` 布尔标志：
   - 若倒计时到了由 AI 自动推选，`isAuto === true`，提示文案展示：“💡 【主持人自主选定了下一步论证方向】”。
   - 若中途由于人类移入鼠标打断倒计时并手动点击了提交，`isAuto === false`，提示文案展示：“【已决定下一步的论证方向】”。
3. **连锁限制保护**：如果检测到 `autonomy === "autonomous"`，在讨论执行中强制置 `enableInquiryLoop = false`。

### 5.3 会议最终报告渲染与归档 (Resolution Report Display)
- 废除“人类主导多维权衡决策看板”分栏（不再聚合展示 decisions, tradeoffs 与 concerns，去除“通过/驳回整改”的状态底栏）。
- **恢复原版 UI 渲染**：会议结束并得出结论后，最终报告展示区恢复为未引入决策面板之前的纯 Markdown 结论气泡卡片渲染样式（支持一键编辑、保存与解锁重启）。

## 六、 体验提升与异常重构 (UX & Edge Cases)

1. **高卡顿调用加载卡片 (Loading state)**：
   由于调用大模型进行判定及选项生成存在一定延迟，系统须在以下两个节点挂载明显的 Loading 态组件：
   - **信息追问判定中**：当向后端 `/api/discussions/inquiry` 请求判定是否需要追问时，页面底部应呈现“主持人正在研判输入信息是否足够，请稍候...”的微磨砂 Loading 卡片。
   - **决策选项提炼中**：当主持人流式总结播放完毕、正在向后端 `/api/discussions/decision-options` 请求提炼方向时，页面底部应呈现“主持人正在提炼下一步备选方向，请稍候...”的 Loading 占位状态。
2. **滚动自动跟随失效防断机制**：
   放宽自动跟随的阈值范围至容器底部 150px 以内，解决 KaTeX 复杂公式、Markdown 异步解析布局重排引起的滚动失联；并在关键布局切换处引入 double-frame 延迟沉底，确保会议流完美跟随。
3. **AI 遗漏控制标签**：如果大模型在主动流控中没有输出 `<next_expert>` 标签，前端将自动采用回退（Fallback）算法：从尚未发言的 `remainCandidates` 列表中取出第一位，强行拉起讨论，防止会议流程死锁。
4. **用户拒绝补充信息（跳过追问）**：在追问面板上，系统提供一个【跳过（Skip）】按钮。如果用户拒绝补充或不知道答案，点击【跳过】可让 AI 按照默认缺省值继续推流讨论。
