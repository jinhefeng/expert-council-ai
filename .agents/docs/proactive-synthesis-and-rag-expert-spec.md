# 主动流式主持人与 RAG 知识外挂及信息索取环《功能与交互说明书》

本文档基于“人类主导、AI 深度参与引导评审”的设计哲学，定义了 `Design Council AI` 系统中外部 RAG 数据库对接、流式主动引导主持人、以及人机信息索取环的业务交互与技术规范。

---

## 一、 功能 A：RAG 专家私有知识库外挂 (External RAG Database Integration)

### 1.1 业务背景
若直接将长篇参考规范（如几百KB的安全准则）全文注入专家的系统提示词，会导致大模型上下文（Context Window）爆炸，不仅带来高昂的 Token 费用，还会引发“Lost in the Middle”的注意力分散效应。因此系统不采用本地文本暴力注入，而应当直接**对接外部已有的 RAG 知识库数据库**。

### 1.2 检索架构与数据模型
系统支持将专家定义与外部向量知识库 API 连接：
1. **数据模型扩展**：
   ```typescript
   export type Expert = TenantScoped & {
     // ... 原有字段
     ragEnabled?: boolean;     // 是否启用外部 RAG 知识检索
     ragEndpoint?: string;     // 外部 RAG 接口地址 (例如 Dify/FastGPT/自研向量服务 API)
     ragToken?: string;        // 外部 RAG API 鉴权 Token
     ragDatasetId?: string;    // 目标知识库/数据集 ID (Collection ID)
   }
   ```
2. **检索流程（API 对接）**：
   - 当触发该专家发言时，后端 API 检测到 `ragEnabled === true` 且配置了 `ragEndpoint`。
   - 向 `ragEndpoint` 发送 HTTP POST 请求，载荷中包含当前会议议题 `{ query: question, datasetId: ragDatasetId }` 并带上 `ragToken` 鉴权首部。
   - 外部 RAG 数据库进行高性能向量计算并返回 Top-K 相关的**文本语义切片（Chunks）**。
   - 后端拼装：将检索到的最相关内容作为客观事实背景，拼接进大模型输入：
     ```text
     【从外部RAG知识库检索到的相关参考规范】：
     {retrievedChunks}
     ```
3. **未配置处理**：若用户未指定外部 API，或外部 API 连接超时，系统不执行任何本地文件降级，直接返回空字符串，专家仅根据原生的 System Prompt 正常发言。

## 二、 功能 B：基于“反馈环状态机”的主动引导与自主决策 (Feedback Loop State Machine)

### 2.1 第一性原理与统一模型定义
从会议评审的本质来看，无论哪种讨论模式，其底层都遵循同一个**“发表 ➔ 冲突 ➔ 局部决议 ➔ 反馈论证 ➔ 一致收敛”**的反馈环路（Feedback Loop）。

系统在底层统一采用**单事件循环状态机**，每轮讨论结束后，程序流都会流经一个核心的中枢点——**“流向抉择点（Steering Hub）”**。三种模式的本质区别，仅仅是该决策点的**“流控抉择算子（Operator）”和触发时机不同**：

```
                    [ 专家发言 (Expert Turn) ]
                               │
                               ▼
               [ 共识整理与待决断项 (Moderator Synthesis) ]
                               │
                               ▼
               ┌────────────────────────────────┐
               │    流向抉择中枢 (Steering Hub)  │
               └───────────────┬────────────────┘
                               │
            ┌──────────────────┼──────────────────┐
            ▼ (算子: 队列自发)  ▼ (算子: 人类用户)   ▼ (算子: AI 主持人)
      [传统模式 (Passive)]    [协调引导 (Facilitative)]  [自主仲裁 (Autonomous)]
      1. 检测剩余候选队列     1. 流程挂起 (Suspend)     1. AI 自动局部决策与指派
      2. 还有专家:            2. 呈现人类协同控制台      2. 自动更新 currentDecisions
         - 自动分发下一人      3. 用户抉择并确认          3. 自主决定是否发起下一轮
      3. 专家发完:               - 做出阶段性决策           - 是: 自动进入 Next Round
         - 主持人总结 ➔ 结束      - 补充新背景信息           - 否: 整合最终结论并退出
                                 - 触发新一轮/结束
```

---

### 2.2 传统被动模式 (Passive Traditional Mode)
- **定位**：专家轮流发言，最后由主持人做单点归纳总结（系统原有基础流）。
- **在统一状态机中的流控表现**：
  1. 系统不更新 `currentDecisions`（决议池在讨论阶段保持为空）。
  2. 当一个专家发言完毕后，流控抉择中枢检测剩余候选专家列表 `remainCandidates`：
     - **若非空**：直接跳过主持人中间插话，指派下一个专家发言，继续 while 循环。
     - **若为空**：触发主持人（AI）流式进行一次性终极总结（Synthesis）。
  3. 主持人流式总结结束后，直接跳出 while 循环，会议宣告结束。

---

### 2.3 协调引导模式 (Human-Steered Facilitative Mode)
- **定位**：人类主导决策，AI 辅助推动。
- **在统一状态机中的流控表现**：
  1. 每轮讨论（例如 2 位专家发言完毕）后，大模型输出本轮共识与分歧，并提示当前需要决断的核心分歧。
  2. **流程进入挂起（Suspend）**，页面底端弹出**“人类协同控制台（Human Steering Console）”**。
  3. 控制台向用户提供三个控制项：
     - **「阶段性决策」**：输入框或单选项（例如：*“看了交互和安全专家的分歧，我决定第一阶段先采用离线登录方案。”*）。
     - **「补充新背景」**：用户可上传新附件或填写补充数据。
     - **「流向选择键」**：
       - 【🟢 触发新一轮论证】：把上述新决议和数据作为上下文，恢复 while 循环，流转至下一位尚未发言的专家对其进行针对性评审。
       - 【🔴 终结会议，提炼结论】：强行清空队列，跳出循环，由主持人基于目前决议起草决议纪要。

---

### 2.4 自主仲裁模式 (AI-Autonomous Arbitration Mode)
- **定位**：AI 主导多轮闭环决策，人类最后过目。
- **在统一状态机中的流控表现**：
  1. 每轮讨论结束后，AI 主持人代表会议直接自主做出局部决策，并输出：
     - 局部决议：`<decision_action>决定采用方案A</decision_action>`
     - 追问指派：`<next_expert>专家C</next_expert>`
  2. **流程不挂起**。系统自动将该 `<decision_action>` 存入已达成决议池 `currentDecisions` 中。
  3. 自主判断是否需要论证：
     - 如果输出了指派标签，while 循环自动启动新一轮。**下一位专家发言时，系统会自动将“主持人当前局部决议”作为前置背景输入**，强力约束该专家对决议的漏洞和风险进行二次表态。
     - 如此多轮反复（直至主持人判定所有专家对决议无新异议，不再输出指派标签；或达到最大轮次保护限制）。
  4. 循环自然结束，AI 主持人输出最终全体达成一致的一致性决议报告，完成闭环。

---

### 2.5 动态调度控制标签协议 (Control Tags Protocol)
为了在流式输出中让主持人安全传递流控信号，大模型将在返回的内容中通过特殊的 XML-like 标签输出控制命令。后端或前端在 stream 接收时将它们拦截并转化为系统事件：
* **指派下一位发言人**：`<next_expert>expert_id</next_expert>`
* **会议终结与表决状态**：`<decision_action>approved | rejected</decision_action>`
* **关键信息追问（索取信息）**：`<inquiry>具体问题</inquiry>`

## 三、 功能 C：人机信息索取与追问环 (Human-in-the-Loop Inquiry Loop)

### 3.1 业务背景
在评审项目（如系统架构、交互原型）时，智能体专家经常由于缺少必要的背景数据（如“当前的QPS上限是多少？”）而只能给出泛化的推测。

### 3.2 会议控制开关 (Inquiry Control Switch)
- **开关设计**：在会议设置中，新增 `enableInquiryLoop?: boolean` 开关（在 UI 上渲染为“启用 AI 追问交互机制”）。
- **分支流控逻辑**：
  1. **开关启用 (`enableInquiryLoop === true`)**：当大模型输出中包含 `<inquiry>` 标签时，自动圆桌的 while 循环**执行挂起（Suspend）**。前端渲染“信息补充面板”，等待用户点击提交数据或点击“跳过”。用户回复后，循环**恢复（Resume）**，并将该补充数据拼入最新的背景上下文。
  2. **开关关闭 (`enableInquiryLoop === false`)**：即使大模型输出了 `<inquiry>` 控制标签，前端在流式解析时也**自动忽略并过滤该标签**。系统不展示追问面板，亦不阻断会议流转，大模型被约束为自我进行缺省推演（并在发言中注明“由于缺少数据，按默认场景评估”），以保障会议流水线一冲到底。

### 3.3 交互与执行流程
当主持人或专家识别到关键背景数据缺失，系统会触发“信息索取环”：

```
[自动圆桌讨论进行中] ── (AI 输出含有 <inquiry> 标签)
       │
       ▼
 1. 自动流程挂起 (Suspend)
    ├─ while 循环暂停，停止向下一个专家分发
    └─ discussesing 状态切换为“等待信息补充”
       │
       ▼
 2. 呈献追问界面 (Inquiry Panel)
    ├─ 在当前消息流底部，以警示框展示 AI 索取的问题
    └─ 提供“补充信息”输入框和发送按钮
```

## 五、 边缘案例处理与平滑降级 (Edge Cases)

1. **AI 遗漏控制标签**：如果大模型在主动流控中没有输出 `<next_expert>` 标签，前端将自动采用回退（Fallback）算法：从尚未发言的 `remainCandidates` 列表中取出第一位，强行拉起讨论，防止会议流程死锁。
2. **用户拒绝补充信息（跳过追问）**：在追问面板上，系统提供一个【跳过（Skip）】按钮。如果用户拒绝补充或不知道答案，点击【跳过】可让 AI 按照默认缺省值继续推流讨论。
3. **知识分块为空**：若 RAG 文档检索出的 Top K 关联度都极低（低于阈值），则不做任何参考文本注入，直接走正常 systemPrompt，确保专家智商不退化。
