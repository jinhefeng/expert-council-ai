# 流式输出与思考过程交互优化说明 (Streaming & Reasoning Interaction Spec)

## 1. 现状痛点分析 (Problem Statement)
目前引入了流式输出（Streaming）与推理模型设置（Reasoning）后，由于底层的状态未进行分离，导致了以下不合理的交互表现：
1. **双重状态呈现（幽灵转圈）**：当上方气泡已经开始逐字输出内容时，由于底部用于标识“正在请求中”的 `speakingExpertIds` 状态并未被解除，导致下方依然挂着一个“正在结合上下文编排论点...”的 Loading 卡片。
2. **纯文本 `<think>` 标签泄露**：针对 DeepSeek-R1 / o1 等深度推理模型，其返回流中会夹杂 `<think>...</think>` 的内部演算逻辑。这些内容目前会被原样扔进 Markdown 解析器，不仅在视觉上没有隔离（混杂在正文中），而且可能因为标签闭合问题导致 Markdown 样式错乱。

## 2. 交互优化设计 (Interaction Design)

### 2.1 请求状态生命周期的精细化拆分
我们将原来粗放的 `discussing` 状态细化为两个子阶段：
- **等待阶段（TTFB, 首字节到达前）**：展示“正在审视议题” Loading 卡片。此时专家刚接到发言指令，正与大模型建立连接。
- **推流阶段（Streaming, 首字节到达后）**：一旦收到第一个推流 Chunk（字），即代表模型开始作答，此时**立刻隐藏底部的 Loading 卡片**，让用户的视觉焦点自然地向上转移到正在生长的 Markdown 气泡框。

### 2.2 推理过程 (Reasoning Process) 的隔离渲染
在前端单条气泡 (`ChatMessage`) 的渲染侧增加纯前端的视图解析器：
1. **实时捕捉 `<think>` 标签**：通过正则匹配 `/<think>([\s\S]*?)(?:<\/think>|$)/`，从 `message.content` 中拦截被 `<think>` 包裹的区域（即使流式过程尚未产生闭合的 `</think>`，也要能贪婪捕获到当前末尾）。
2. **独立 UI 区块**：将捕获出的推理内容，从正文部分剥离，放置在一个具有弱化视觉（如虚线边框、灰色背景、斜体）并带有一个 `🤔 模型思考过程` Title 的可折叠或独立区块中。
3. **正文保持纯净**：剥离后的剩余内容才传入 `ReactMarkdown`，确保最终专家观点的正式性与 Markdown 样式的稳定性。

## 3. 架构考量与潜在影响 (Architecture & Impact)
- **纯视图层操作**：此优化完全在 `src/app/page.tsx` 中完成。
- 对后端的 `expertStance` 结构化截取毫无影响（因为底层的 content 并没有丢掉 `<think>`，只是在渲染那一刻做了隔离）。
- 对于非推理模型（普通 OpenAI），没有 `<think>`，因此正则安全通过，不会触发该逻辑，兼容性良好。
