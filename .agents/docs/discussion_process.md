# 专家圆桌会议讨论全流程梳理

本文档梳理了“智能体圆桌会议中心”从用户输入议题，到多智能体交互，直至最终在界面展示结论的全过程。主要基于 `src/app/page.tsx` 中的核心控制逻辑。

## 1. 触发与初始化 (User Input Phase)
**函数入口：** `handleSubmitDiscussion`

- **收集上下文**：用户在底栏输入框输入议题，点击发送。前端收集用户输入的 `question`，已上传的文件附件列表（转化为 `sources`，若是文本会截取摘要），以及当前设置的 `projectContext`。
- **消息上屏**：立即构建一条 `role: "user"` 的 `ChatMessage`，存入当前会议的 `messages` 数组并持久化到本地存储（LocalStorage），使用户的发言立即在聊天界面上渲染出来。
- **中断控制器创建**：创建一个 `AbortController` 并存入 `discussAbortControllersRef`，以便用户中途随时叫停当前的生成过程。
- **模式分支判定**：
  - 如果未选择任何专家参与，直接跳入第 4 步的主持人总结阶段。
  - 如果当前的发言模式（`turnOrderMode`）设为 `manual`（点名发言），则停止自动流程，等待主持人（用户）手动点击某位专家的“点名发言”按钮 (`handleCallExpertDirectly`)。
  - 若为自动流转（默认的 `sequential` 或 `relevance`），则进入专家流转环节。

## 2. 专家排队与调度 (Expert Turn Scheduling)
**调度循环**：针对当前勾选的所有参会专家，建立候选队列 `remainCandidates`，在队列清空前进行循环流转。

- **确定下一位发言人**：
  - 若模式为 `relevance`（智能相关度）且队列中有多位专家：调用 `/api/discussions/next-speaker`，大模型会根据当前的议题、上下文及历史发言，选出最适合接下来接话的专家 ID。
  - 否则，按默认顺序取队列的第一位。
- **状态更新**：将选出的专家从候选队列中移除，并设置状态 `speakingExpertIds`。这会触发界面上该专家的卡片以及聊天流中显示“正在思考/Speaking...”的骨架屏（Loading 动画）。

## 3. 专家观点生成 (Expert Generation)
**核心请求：** `requestExpertTurn`

- **API 调用**：前端向 `/api/discussions/expert-turn` 发送 POST 请求，携带当前议题、项目背景、该专家的角色设定（包括其对抗强度 `debateIntensity`）、本轮之前其他专家的发言（`previousTurns`）以及整场的历史对话 `conversationHistory`。
- **界面渲染**：API 返回该专家的论述内容以及结构化的立场数据（`expertStance`，包含：立场观点、关键风险、实施建议、方案取舍）。
- **消息上屏**：构造一条 `role: "expert"` 的 `ChatMessage`，推入会议的 `messages` 中并立即保存。UI 层渲染该专家的 Markdown 内容并在下方展示黄色的“结构化卡片”。

*(循环步骤 2 和 3，直至候选队列中的所有专家都完成发言)*

## 4. 主持人全局汇总 (Moderator Synthesis)
**核心请求：** `requestSynthesis`
当所有专家发言完毕后，系统自动交接给“主持人”进行总结。

- **状态更新**：清空专家的思考状态，将 `synthesisPendingMeetings` 设为 `true`，聊天流中出现“主持人正在汇总本轮会议纪要”的动画提示。
- **API 调用**：调用 `/api/discussions/synthesis`。大模型作为主持人，综合本轮所有专家的观点，提取出：`consensus`（共识）、`disagreements`（分歧）、`decisions`（决议）、`nextActions`（后续行动）以及一段总体总结陈词。
- **消息上屏**：构造一条 `role: "moderator"` 的 `ChatMessage`，结构与专家发言类似，展示在聊天流的最底端，标志着本轮讨论的闭环。

## 5. 最终结论提炼与展现 (Final Conclusion)
**函数入口：** `handleGenerateConclusion`
当整场会议经过多轮讨论，用户认为信息已经充分时，可以触发提炼最终结论。

- **生成过程**：前端设置 `generatingConclusions` 为 `true`，界面出现“正在提炼最终结论”的胶囊动画提示。
- **API 调用**：向 `/api/discussions/conclusion` 传递全部历史对话内容，让大模型撰写一份排版精良的 Markdown 终版总结报告。
- **状态落盘**：请求返回后，将这段文本保存到当前会议的 `finalConclusion` 字段中。
- **界面展示与编辑**：
  - 聊天界面底部会展示一个金黄色的“会议最终结论”面板。
  - 面板渲染生成的 Markdown。
  - 面板右上角提供【编辑】按钮，用户点击后可进入编辑模式（`isEditingConclusion`），在 `textarea` 中手工微调，并点击保存。

## 6. 大模型交互机制与上下文隔离 (LLM Interaction & Isolation)
系统通过极其精细的提示词工程 (Prompt Engineering) 和历史记录清洗机制，在同一个大模型 API 接口下实现了多位智能体的独立运转。

### 6.1 角色专属的系统提示词注入 (System Prompt)
在每次请求发起前，系统会组装一段动态的、只属于当前指定专家的 `System Prompt`。大模型每次接到的请求都是完全独立的无状态 HTTP POST（绝不复用 Session 或上下文 ID），这从物理链路上杜绝了互相污染。系统提示词的构成如下：

1. **基础人设与特质**：直接读取预设的（或用户自定义的）专家资料。
   - 示例：“你的性格与气质：务实、理性、关注落地转化。”
   - 示例：“你关注的焦点：用户留存、核心链路转化率、商业闭环。”
2. **动态对抗强度注入 (`getIntensityPrompt`)**：系统将该专家的个人对抗倾向与当前会议全局对抗强度取均值，计算出当前的辩论强度 `Level`，并动态拼接对应的行为指示：
   - **Level 1-2（温和协作）**：“在圆桌会议中，你应当温和、包容，倾向于寻找共识。请重点指出你同意之前发言专家的哪些意见，并在此基础上做建设性的微调与补充。避免激烈的观点对抗，语气要柔和友好。”
   - **Level 3（中立理性）**：“在圆桌会议中，你应当客观、理性地表达专业判断。不用刻意迎合，也无需刻意抬杠。如实指出你的视角关注的核心问题，并提出独立的专业建议。”
   - **Level 4-5（激烈批判）**：“在圆桌会议中，你代表此视角的极致挑剔立场。态度必须极其强硬、敏锐，充满批判性。请主动挑战并质问前几位发言专家的漏洞，无情揭示其方案逻辑或盲区……请用犀利、直接的语气表态，并给出强烈的反制或替代建议。”
3. **第一人称锚定与输出约束**：
   ```text
   请以第一人称（我是某某某）的口吻直接输出你的发言。
   IMPORTANT: 必须全程使用中文（简体中文）进行回答！
   输出要求：
   1. 包含一段直观生动的会议发言内容（content）。
   2. 在发言的最后，必须提供一个纯 JSON 格式的结构化摘要（便于前端拆分展示），包含 stance, concern, recommendation, tradeoff。
   ```
   这种强力的第一人称约束能让大模型在一瞬间精确“穿上”对应角色的外衣。

### 6.2 历史上下文的清洗与传输 (`formatConversationHistoryForLLM`)
前端将整场的历史聊天记录 `ChatMessage[]` 传递给后端，后端将其统一转换为大模型兼容的 `[{role, content}]` 数组形式，其中运用了关键的防污染技巧：

- **用户提问映射**：真实用户的原始提问被直接映射为 `role: "user"`。
- **防止上下文污染的杀手锏**：以往所有的专家或主持人的发言全部被映射为 `role: "assistant"`。由于在同一段历史中存在多条不同角色的 `assistant` 记录，大模型极易产生身份错乱，把别人的发言当成是自己刚才说的。为解决此问题，后端会强制在每一条 `assistant` 记录的内容开头拼接说话人的身份标识：
  - 转换前：`"我觉得这个设计需要修改。"`
  - 转换后：`"【交互设计专家 (总监)】：我觉得这个设计需要修改。"`
  
  当大模型带着前文（6.1）设定的“我是产品经理”的 `System Prompt` 读到这段历史时，它能在逻辑上极其清晰地感知到：“这句 `assistant` 历史记录是我的同事交互专家说的，而我现在扮演的是产品经理”，从而完美规避“串台”或“抢词”现象。

### 6.3 本轮最新上下文的组装
在完成了 System Prompt 和清理过的历史记录拼接后，为了促成最新轮次的回答，后端会在数组最后追加一条全新的 `user` 角色消息，把当前的最新状况打包喂给大模型：

```text
当前会议新议题：{question}

项目背景及附件信息：
{projectContext_and_attachments}

本轮专家讨论中，此前已发言记录：
【品牌专家】：我认为品牌调性需要...
【研发专家】：技术上这个目前不可行...

请针对当前议题表达你的专业视角发言：
```
这段聚合信息确保了该专家能精准把握：1. 我们当下在聊什么议题；2. 会议的客观背景是什么资料；3. 刚才在同一轮里，前面几个人说了什么（方便顺承或反驳）。

## 附加：中断与导出机制
- **手动叫停 (`handleAbort`)**：任何处于生成中的网络请求，均可通过其绑定的 `AbortController` 立即截断。界面上会追加一条由“系统提示”发送的红字告警消息，告知用户讨论已被手动中止。
- **导出记录 (`handleExportToHTML`)**：系统提取当前聊天容器的 DOM 以及所有的 CSS 样式，拼接成一个独立的 `.html` 文件，自动触发浏览器下载，实现所见即所得的离线留档。
