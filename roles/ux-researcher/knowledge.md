# UX 研究员知识底座

## Canonical Books

- Don Norman: *The Design of Everyday Things*
  - 核心摘要：可理解性、反馈、映射和约束会影响用户能否正确行动。
- Steve Krug: *Don't Make Me Think*
  - 核心摘要：界面应降低理解成本。评审时要检查用户是否需要额外思考才能找到下一步。
- Alan Cooper et al.: *About Face*
  - 核心摘要：目标导向设计强调用户目标、场景和行为模型。评审时要区分用户目标和系统功能。
- Louis Rosenfeld, Peter Morville, Jorge Arango: *Information Architecture*
  - 核心摘要：组织、标签、导航和搜索决定信息是否可被理解和找到。
- Steve Portigal: *Interviewing Users*
  - 核心摘要：研究要理解真实行为、动机和上下文，而不是只收集意见。
- Erika Hall: *Just Enough Research*
  - 核心摘要：轻量研究也能降低决策风险。评审时要提出可执行的验证方式。
- Bella Martin, Bruce Hanington: *Universal Methods of Design*
  - 核心摘要：不同研究方法适合不同阶段，应该根据问题选择方法。

## Methods

- Task flow review
  - 适用：判断用户从进入页面到完成目标是否有清晰路径。
- Cognitive load audit
  - 适用：检查信息密度、术语、选项数量和视觉噪音。
- Jobs-to-be-done framing
  - 适用：把用户问题从“想要功能”还原成“想完成的进展”。
- IA label check
  - 适用：检查导航、分组、按钮和标题是否符合用户语言。
- Assumption mapping
  - 适用：把高风险、低证据的判断列出来，转为验证计划。
- Usability test script
  - 适用：为关键流程设计 3-5 个任务，观察用户是否能独立完成。

## Domain Heuristics

- 如果用户必须先理解公司内部术语，信息架构通常有问题。
- 如果页面同时要求用户做多个重要选择，决策成本会升高。
- 如果 CTA 文案只描述系统动作，不描述用户收益，下一步会变弱。
- 如果用户不知道自己处在流程哪一步，信任和完成率都会下降。
- 如果设计依赖解释文字才能成立，交互本身可能不够清楚。

## Source Use

这些来源只作为方法框架，不直接复制原文。回答时应把结论标成“判断”或“假设”，并给出可验证的下一步。
